## Scope Boundaries

This design is an umbrella flow. Implementation should be split by schema:

| Schema | Boundary | Files |
|--------|----------|-------|
| `world-yaml` | Defines and maintains book data. No engine or TUI behavior. | `src/core/types.ts`, `src/core/schemas/content-pool.ts`, `src/core/content-pool-loader.ts`, `src/core/world.ts`, `worlds/content-pool/<domain>.yaml` |
| `world-engine` | Reads validated book data and returns a command result. No UI rendering. | `src/engine/player-actions.ts`, `src/engine/command-executor.ts`, `src/engine/capability-provider.ts` |
| `world-tui` | Displays `bookDisplay` returned by the server. No engine imports. | `src/shared/protocol.ts`, `src/tui/client/game-client.ts`, `src/tui/key-layer/index.ts`, `src/tui/app.tsx`, `src/tui/panels/book-reader/book-reader-panel.tsx` |

Do not keep this as an engine-only change. The engine can only be validated after `bookContents` is typed, loaded, and available in the ContentPool.

## ContentPool Integration

### New data model

```ts
export interface BookContent {
  id: string;
  itemTemplateId: string;
  title: string;
  pages: string[];
}
```

Zod schema:

```ts
export const BookContentSchema = z.object({
  id: z.string().min(1),
  itemTemplateId: z.string().min(1),
  title: z.string().min(1),
  pages: z.array(z.string().min(1)).min(1),
});
```

ContentPool field:

```ts
bookContents: BookContent[];
```

Book text is LLM-evolvable content. A readable item can be generated by LLM world evolution, so the same change must provide a write tool and a persisted mutation path. Do not rely on prompt-only instructions; tool definition, parsing, materialization, and evolve YAML write-back must all exist.

### Maintenance path

| Checklist area | Required decision |
|----------------|-------------------|
| Type | Add `BookContent` and `ContentPool.bookContents`. |
| Schema | Add `BookContentSchema`; include `bookContents` in `ContentPoolSchema`. |
| Loader domain | Register `bookContents` in `DOMAIN_FIELDS`; choose one YAML domain such as `books` or `lore-books`. |
| Domain schema | Register the domain in `DOMAIN_SCHEMAS` so YAML load failures are visible. |
| Default content | Add `bookContents: []` or baseline seed content in `createDefaultContentPool()`. |
| YAML content | Store shipped book text in `worlds/content-pool/<domain>.yaml`, including `sutra_copy`. |
| Mutation type | Add `addBookContents?: BookContent[]` to `ContentPoolMutation`. |
| Tool schema | Add `ADD_BOOK_CONTENT_TOOL` in `src/llm/tools/content-pool-evolve.ts`. |
| Tool parser | Parse `add_book_content` in `src/llm/tool-mutations.ts` with `BookContentSchema`. |
| Materializer | Apply `addBookContents` in `src/simulation/content-pool-materializer.ts`. Replace existing content by `id`, or by `itemTemplateId` when ids differ but the item already has a book. |
| Prompt | Update `buildContentPoolEvolvePrompt()` so the LLM knows readable items require matching book content. |
| Evolve write-back | Add `bookContents` to `writeEvolveDeltas()` for the selected book domain. |
| Consumer | Engine reads through `world.contentPool.bookContents` only after the field exists. |
| Tests | Add loader/schema/tool/materializer/write-back tests proving book content validates, loads, can be generated by tool call, and persists to evolve YAML. |


Tool rule:

| Situation | Required LLM behavior |
|-----------|-----------------------|
| LLM creates a readable item template | Also call `add_book_content` for the same `itemTemplateId`. |
| LLM updates an existing readable book | Reuse the same `id`; replace pages intentionally. |
| LLM creates a non-readable item | Do not call `add_book_content`. |

The materializer should reject or log invalid tool calls instead of creating partial readable content.

### Reads (new field тАФ from dependent world-yaml change)

| pool.xxx | Where | Why |
|----------|-------|-----|
| `bookContents` | `command-executor.ts` `executeRead()` | Lookup book text (title + pages) by item `itemTemplateId` |

### Missing content behavior

A readable shipped or LLM-generated item without matching `bookContents` is invalid content. Do not hide that with a generic book-reading fallback in engine code.

Recommended behavior:

| Situation | Behavior |
|-----------|----------|
| Item is not readable | Return a normal command error. |
| Item is readable but no matching book content exists | Return a content error and do not apply read effects. Add a test so shipped content cannot silently miss book text. |
| Item is readable and book content exists | Return `bookDisplay` and apply read effects through the delta pipeline. |

## State Mutation Path

All state mutation goes through the standard delta pipeline:

```
executeRead() тЖТ returns SimulationDelta
      тЖУ
executeEntityAction() тЖТ composeDeltas(actionDelta, rippleDelta)
      тЖУ
applyDelta(world, mergedDelta)
  тФЬтФАтФА needChanges тЖТ entity.needs[type].value += delta      (EXISTING path)
  тФФтФАтФА traitModifiers тЖТ entity.traits modify/replace          (EXISTING path, see applyDelta)
```

Verify that `applyDelta()` already handles `traitModifiers`:

- `SimulationDelta.traitModifiers` is an existing field in `delta-composer.ts` (line 106, 128)
- `applyDelta()` in `src/core/world.ts` already processes `traitModifiers`

So NO new mutation path needed тАФ `traitModifiers` flows through the existing pipeline.

### Item non-consumption

`executeRead()` does NOT call `inventory.splice()` or `world.entities.delete()`. Unlike `executeEat()` (which consumes the item), `executeRead()` leaves the item untouched.

## TUI Reader Design

### Protocol surface

Add a shared protocol type so client and server agree on the payload:

```ts
export interface BookDisplay {
  title: string;
  pages: string[];
}

export interface CommandResultMessage {
  type: "command_result";
  events: CommandEvent[];
  delta?: SimulationDelta;
  ended: boolean;
  bookDisplay?: BookDisplay;
}
```

Server already sends `{ type: "command_result", ...result }`, so the TUI work mainly needs the shared type and client handling.

### Client state

Add a `BookReaderState` to `src/tui/client/game-client.ts`:

```ts
interface BookReaderState {
  title: string;
  pages: string[];
  pageIndex: number;
}
```

Expose these client methods:

| Method | Behavior |
|--------|----------|
| `bookReader()` | Returns current reader state or `null`. |
| `openBookReader(bookDisplay)` | Sets `{ title, pages, pageIndex: 0 }` and pushes `book-reader` layer. |
| `closeBookReader()` | Clears state and pops `book-reader` layer. |
| `nextBookPage()` | Moves to next page if one exists. |
| `prevBookPage()` | Moves to previous page if one exists. |

On `command_result`, after pushing events and item notifications, call `openBookReader(message.bookDisplay)` when present. Keep event log feedback; the reader is an additional panel, not a replacement for command feedback.

### Panel layout

Add `src/tui/panels/book-reader/book-reader-panel.tsx`.

Behavior requirements:

| Requirement | Detail |
|-------------|--------|
| Paging | `pages[pageIndex]` is the only visible page body. Do not render all pages in one scrollbox. |
| Boundaries | At first page, previous-page action is disabled/no-op. At last page, next-page action is disabled/no-op. |
| Closing | `Esc` closes the reader and returns to the previous layer. |
| Responsive size | Use `getModalMetrics()` like other popup panels; keep the page body wrapped and clipped inside the panel. |
| Empty data | Empty pages should be rejected by schema, so the panel should not need a runtime empty-content fallback. |
| Text source | Title and page text come from server payload. Do not hardcode book text in TUI. |

### Key layer

Add `BOOK_READER_LAYER` in `src/tui/key-layer/index.ts`:

| Key | Action |
|-----|--------|
| `escape`, `q` | Close reader. |
| `left`, `h`, `pageup` | Previous page. |
| `right`, `l`, `pagedown`, `space` | Next page. |

Register the layer in `ALL_LAYERS` and mount `<BookReaderPanel />` in `src/tui/app.tsx` next to other popup panels.
