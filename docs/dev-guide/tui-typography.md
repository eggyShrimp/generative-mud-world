# TUI 文本排版规范

本文定义 MUD 文本、列表、日志和提示语的排版规则。终端 UI 的文字是主要画面，因此排版必须稳定、清楚、可扫读。

## 基本原则

- 文本优先可读，不追求花哨。
- 中文、数字、符号混排时，要考虑显示宽度。
- 列表项要能快速扫读。
- 日志前缀必须对齐。
- 提示语要短，不解释功能原理。

## 房间文本

房间文本包含房间名和描述。

规则：

- 房间名单独一行。
- 描述使用自然句，不用项目符号。
- 描述允许自动换行。
- 不在房间描述中混入可操作菜单。
- 房间描述下方先显示出口，再显示目标。

示例：

```text
灰鸦旅店
木梁被烟熏得发黑，柜台后传来酒杯碰撞的声音。雨水顺着门缝渗进来，在地上留下一道暗痕。
```

## 出口列表

出口是高频操作，必须清楚可见。

格式：

```text
[N] 北  · 大道 · 1格
[E] 东  · 小径 · 2格
```

规则：

- 快捷键放在最前。
- 方向使用短中文。
- 地形和距离是补充信息，用分隔点连接。
- 没有出口时显示 `无出口`。
- 不把出口混进目标列表。

## 目标列表

目标列表显示当前房间可交互对象。

格式：

```text
[1] 旅店老板        人物
[2] 铜币            物品
```

选中格式：

```text
> [1] 旅店老板      人物
```

规则：

- 编号固定在最前。
- 目标名在中间。
- 类型在右侧。
- 选中态只加 `>`，不改变列表结构。
- 没有目标时显示 `眼前没有可交互目标。`

## 动作菜单

动作菜单用于目标动作弹窗。

格式：

```text
[1] 交谈
[2] 观察
Esc 返回
```

规则：

- 一行动作只放一个快捷键。
- 不在动作旁解释功能。
- 动作名使用动词。
- `Esc 返回` 固定放在底部。

## 事件日志

事件日志每行由两列组成：

```text
前缀列  正文列
```

示例：

```text
→   你向北走去。
✦   老板说：“天快黑了。”
✓   获得 铜币。
!   这里没有这个出口。
```

规则：

- 前缀列固定显示宽度。
- 禁止直接用 `prefix + " " + text` 拼接日志。
- 优先用横向布局把前缀列和正文列分开渲染。
- 如果必须字符串拼接，必须使用显示宽度计算或稳定的制表规则。
- 少用 emoji，因为不同终端的宽度差异很大。
- 正文自动换行时，从正文列开始换，不从前缀列开始。

## 日志前缀

推荐前缀：

| 类型 | 前缀 |
|------|------|
| 移动 | `→` |
| 获得 / 丢下 / 使用 | `✓` |
| 观察 | `?` |
| 休息 / 等待 | `~` |
| 状态 | `●` |
| 背包 | `■` |
| 对话 | `✦` |
| 错误 | `!` |
| 系统 | `·` |
| 结算 | `◆` |
| 世界事件 | `◆` |

## 状态条

需求条格式：

```text
精力  ███████░░░  72
饥饿  ███░░░░░░░  30
安全  █████████░  91
```

规则：

- 标签固定宽度。
- 条长度固定。
- 数字使用整数。
- 低状态用危险色，中间状态用对话色，高状态用成功色。

## 提示语

提示语要短，偏游戏内表达。

推荐：

- `行动中`
- `世界正在回应...`
- `等待回应...`
- `Esc 返回`
- `背包是空的。`
- `无出口`

避免：

- `请等待服务器返回结果。`
- `正在处理操作请求。`
- `当前无可用 capability。`

## 验收清单

- 日志前缀是否对齐。
- 中文和符号混排是否没有明显错位。
- 列表是否能快速扫读。
- 出口是否比普通正文更容易发现。
- 弹窗文字是否短。
- 提示语是否避免工具化表达。
- 宽屏和窄屏是否都没有文字互相覆盖。
